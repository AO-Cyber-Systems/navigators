<gotchas>
- Eden's SignUp flow creates a company per user. For admin-created users, do NOT call SignUp. Instead: hash password, create user directly in auth store, create membership in MaineGOP company, assign role. This requires access to the underlying pgstore methods.
- Eden's auth store uses transactions (TxAuthStore). Admin user creation must use the same transactional pattern for atomicity.
- Password reset must ALWAYS return the same response ("If an account exists, a reset link has been sent") regardless of whether the email exists (pitfall 4 from research).
- Eden's RBAC enforcer checks HasPermission(ctx, userID, companyID, "feature:action"). The permission string is "feature:action" concatenated with a colon.
- Eden seeds system roles in migration 003_roles_permissions. The navigators seed migration must NOT re-create these roles -- just seed navigators-specific permissions and map them to the existing system roles.
</gotchas>
