<objective>
Build the voter data model (migrations) and complete import pipeline for Maine CVR and L2 voter files.

Purpose: Admins need to upload voter registration files from the Maine Secretary of State (CVR, pipe-delimited) and from L2 vendor (tab-delimited CSV). The system must parse these files, load them into a staging table, validate, normalize, generate dedup keys, and UPSERT into the production voters table. Import progress and error reporting must be tracked.

Output: Voters table migration, import jobs/staging migration, proto definitions for import service, CVR and L2 parsers, import orchestration service, ConnectRPC handlers, sqlc queries.
</objective>

<anti_patterns>
- DO NOT store full date of birth -- only year_of_birth INT per Maine 21-A s196-A
- DO NOT parse or store SSN, driver's license, felony/criminal fields even if present in source files
- DO NOT load entire file into memory -- stream-parse line by line with bufio.Scanner (CVR files can be 500MB+)
- DO NOT use single-row INSERTs in a loop -- use pgx CopyFrom for staging (10-100x faster)
- DO NOT geocode synchronously during import -- return import results immediately, geocode async (TRD 02-02)
- DO NOT build dedup on full address string -- normalize components separately
- DO NOT create GIN trigram indexes before bulk import -- they slow bulk inserts massively; create in migration but note that future optimization may DROP/RECREATE around imports
</anti_patterns>

<error_recovery>
- If CopyFrom fails mid-batch: the staging table has import_job_id FK, so DELETE FROM import_staging WHERE import_job_id = $1 cleans up. Update import_jobs.status = 'failed' with error details.
- If UPSERT merge fails: staging data is preserved for retry. Update job status to 'failed', log the SQL error in import_jobs.errors JSONB.
- If file parsing fails partway: track line_number in staging rows. ParseErrors are accumulated (not fatal). Lines with parse errors get is_valid=false with validation_error populated.
- If MinIO presigned URL expires before upload completes: client gets 403 from MinIO. Frontend should request a new presigned URL. Import job stays in 'pending' until file is confirmed.
</error_recovery>

<research_context>
From 02-RESEARCH.md:
- CVR files are pipe-delimited text. Field order is NOT standardized -- build configurable field mapper (admin maps columns on first import). Persist mapping per source_type in import_jobs.field_mapping JSONB.
- L2 files are tab-delimited with 600+ fields. Use encoding/csv with Comma = '\t'. Same configurable field mapper.
- Staging table pattern: COPY raw data into import_staging, validate/normalize, then INSERT ON CONFLICT to merge into voters.
- Dedup key: UPPER(last_name) || '|' || street_number || '|' || UPPER(normalized_street_name) || '|' || zip5 || '|' || year_of_birth
- pgx CopyFrom for bulk staging (10-100x faster than individual INSERTs for >1000 rows)
- Voting history stored as JSONB array: [{"election": "2024-11-05 General", "voted": true, "method": "in_person"}]
- Maine party codes: D, G, L, NL, R (or unenrolled/blank)
- Eden upload.Service provides presigned PUT URLs via RequestUploadURL(ctx, companyID, uploaderID, filename, contentType, sizeBytes) and CompleteUpload(ctx, attachmentID, uploaderID). MaxFileSize = 100MB. Need to add "text/csv" and "text/plain" to allowed content types or use the MinIO client directly for voter files which can exceed 100MB.
</research_context>

<gotchas>
- Eden upload.Service has MaxFileSize = 100MB and a whitelist of content types. Voter files (especially statewide CVR) can exceed this. Use MinIO client directly for voter file uploads rather than eden upload.Service. Create a dedicated "voter-imports" bucket.
- sqlc.yaml overrides geometry to string. When inserting PostGIS POINT values, use ST_SetSRID(ST_MakePoint(lon, lat), 4326) in SQL. The Go side passes lon/lat as floats, not geometry strings.
- The import_staging table uses BIGSERIAL (not UUID) for id since it's a high-volume temp table.
- Census Geocoder coords are "lon,lat" (longitude first, then latitude) -- opposite of typical lat/lng order.
- sqlc needs all referenced tables in schema path. eden migrations are already in sqlc.yaml schema list so FKs to companies(id) and users(id) resolve.
</gotchas>
