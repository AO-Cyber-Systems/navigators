<objective>
Set up the Drift local database with SQLite3MultipleCiphers encryption, define all local tables (voters, contact_logs, sync_operations, sync_cursors, turf_assignments), create server-side sync pull endpoints, and implement the initial data download flow.

Purpose: This is the foundation for offline-first. Without an encrypted local database and a way to pull data from the server, nothing else in the sync engine works.

Output: Encrypted Drift database on Flutter, server sync proto + handler, cursor-based pull sync for voters and contact logs, DAOs for local queries.
</objective>

<anti_patterns>
- DO NOT use sqlcipher_flutter_libs -- DEPRECATED as of drift 2.32.0. Use SQLite3MultipleCiphers via pubspec hooks
- DO NOT use NativeDatabase(File(...)) -- use driftDatabase(name:, native: DriftNativeOptions(...))
- DO NOT use streaming RPCs for sync -- use batched unary RPCs with cursor pagination
- DO NOT store encryption key in SharedPreferences -- use flutter_secure_storage
- DO NOT re-download entire voter dataset on every sync -- use cursor-based delta
- DO NOT use moor/moor_flutter -- old names for Drift, all APIs renamed
</anti_patterns>

<error_recovery>
- If `PRAGMA cipher` returns empty, SQLite3MultipleCiphers is not bundled. Check pubspec.yaml hooks section has `sqlite3: source: sqlite3mc`
- If WorkManager background task can't access DB, check `shareAcrossIsolates: true` in DriftNativeOptions
- If cursor becomes invalid (server data migration), server should return `cursor_valid: false` and client resets to full sync
- If initial download fails mid-batch, preserve progress via cursor -- next attempt resumes from last cursor
</error_recovery>

<research_context>
From 04-RESEARCH.md:
- Drift 2.32+ with drift_flutter 0.3+: use `driftDatabase(name:, native: DriftNativeOptions(shareAcrossIsolates: true, setup: (rawDb) { rawDb.execute("PRAGMA key = '...'"); }))`
- SQLite3MultipleCiphers bundled via pubspec hooks, NOT a pub dependency: `hooks: user_defines: sqlite3: source: sqlite3mc`
- Encryption key: generate random key on first login, store in flutter_secure_storage. If user logs out, key cleared and DB deleted.
- Cursor-based pull: each entity type has a sync cursor (ISO timestamp). Server returns `next_cursor` and `has_more` for pagination.
- Batch size 500 for voter pull to avoid blocking UI.
- Use `database.batch()` for bulk inserts (single transaction, much faster than individual).
- PRAGMA journal_mode=WAL and PRAGMA foreign_keys=ON in beforeOpen.
</research_context>
