String _generateUuid() {
  final rng = Random.secure();
  final bytes = List<int>.generate(16, (_) => rng.nextInt(256));
  bytes[6] = (bytes[6] & 0x0f) | 0x40;
  bytes[8] = (bytes[8] & 0x3f) | 0x80;
  final hex = bytes.map((b) => b.toRadixString(16).padLeft(2, '0')).join();
  return '${hex.substring(0, 8)}-${hex.substring(8, 12)}-...-${hex.substring(20)}';
}

<anti_patterns>
- Do NOT use a phone state detection plugin (flutter_phone_call_state, phone_state). Use a manual "Call Ended" button. Research explicitly rejected these due to permissions overhead and unreliability.
- Do NOT block UI until call ends. The native dialer takes focus; when user returns, the script screen should still be visible.
- Do NOT try to detect the exact moment the call ends. The PhoneCallScreen persists in the nav stack.
- Do NOT use the uuid package for generating UUIDs. Use the existing dart:math Random.secure() _generateUuid() pattern.
- Do NOT use rxdart. Use custom StreamController merging pattern from Objective 05.
- Do NOT use EdenTimeline (does not exist). Use custom ListView pattern from contact_timeline_widget.dart.
- Do NOT use EdenFormWizard. Use step-based flow with local enum state (same as door knock screen).
</anti_patterns>

- If call script is null (not synced yet): show "No script available" with EdenEmptyState and still allow proceeding to disposition without script.

<research_context>
- Click-to-call: `launchUrl(Uri.parse('tel:$phone'))` via url_launcher (already in pubspec ^6.3.0)
- Android 11+ needs `<queries><intent><action android:name="android.intent.action.DIAL"/><data android:scheme="tel"/></intent></queries>` in AndroidManifest.xml
- Phone call flow: launch dialer -> show script during call -> manual "Call Ended" button -> disposition -> sentiment -> notes -> save
- CallDisposition enum: answered/voicemail/no_answer/refused/busy (5 values vs door knock's 4)
- Non-answered dispositions quick-save and pop (same as door knock pattern)
- Only answered disposition gets sentiment + notes step
- Call script variable interpolation: {{voter.firstName}}, {{voter.lastName}}, {{voter.party}}, {{voter.address}}, {{voter.lastContact}}
</research_context>

<gotchas>
- AndroidManifest.xml: Add tel: intent query INSIDE the existing `<queries>` block (after the PROCESS_TEXT intent), not as a separate `<queries>` element
- canLaunchUrl returns false on iOS Simulator (no Phone app). Don't gate the disposition flow on successful call launch.
- The voter.phone field may be empty -- hide/disable call button when phone is null or empty
- PhoneCallScreen flow is slightly different from DoorKnockScreen: it has a "calling" step showing script BEFORE disposition (DoorKnock goes straight to disposition)
- Steps for phone: calling (script display) -> disposition -> sentiment (if answered) -> notes (if answered) -> save
- ContactLogsCompanion uses contactType: 'phone' (not 'door_knock'), and doorStatus stores the phone disposition value (answered/voicemail/no_answer/refused/busy)
- The timeline service already handles contactType 'phone' with a blue phone icon. Enhancement is adding disposition-specific labels like "Phone Call - Voicemail", "Phone Call - Answered" etc.
</gotchas>
