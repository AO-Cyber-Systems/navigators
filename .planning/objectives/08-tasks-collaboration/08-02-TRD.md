<anti_patterns>
- Do NOT poll database for progress updates -- use NATS event from contact_log.created (research anti-pattern)
- Do NOT use deprecated SendAll/SendMulticast -- use SendEach/SendEachForMulticast (research State of Art)
- Do NOT store FCM tokens in navigators tables -- use eden's device_tokens table (research anti-pattern)
- Do NOT assume single device per user -- always send to ALL tokens via SendEachForMulticast
- Do NOT use increment/decrement for progress -- use SQL-based recalculation to avoid race conditions (research Pitfall 3)
</anti_patterns>

<error_recovery>
- If Firebase init fails: log warning and continue (like NATS pattern) -- push notifications disabled but server runs
- If FCM send fails for specific tokens: check IsRegistrationTokenNotRegistered, delete stale tokens
- If NATS consumer loses connection: JetStream durable consumer auto-reconnects; messages redelivered
- Progress calculation race: SQL subquery recalculation is idempotent -- safe for concurrent events
</error_recovery>

- FCMDispatcher: implements eden `notification.Dispatcher` interface (SendPush + IsEnabled)

- NATS stream: "NAVIGATORS_TASKS" with subjects "navigators.task.>" and "navigators.contact_log.>"
- Contact log event: publish in sync_service.go processContactLog() after successful UpsertContactLogFromSync
- Task progress recalculation: SQL UPDATE from subquery counting task_voters (idempotent, race-safe)
- Eden pgstore.Backend does NOT expose NotificationStore -- use eden's generated db.Queries directly for device token operations, or create a simple adapter
- Task reminder: Go ticker that queries tasks with due_date within 24h, dispatches FCM
</research_context>

- Background handler in Flutter must be top-level function with @pragma('vm:entry-point') -- but that's TRD 03 concern

    ```
      js        jetstream.JetStream

      cancel    context.CancelFunc

    Start(ctx):
      1. Create/update stream with subjects: ["navigators.task.>", "navigators.contact_log.>"]
      2. Create durable consumer "task-progress-worker" filtering "navigators.contact_log.created"
      3. Start consume goroutine: on each msg, call processContactLogCreated
      4. Start reminder ticker: every 1 hour, query GetTasksDueSoon, for each task publish "navigators.task.reminder" event
      5. Create durable consumer "task-notification-worker" filtering "navigators.task.>"
      6. Start consume goroutine: on each msg, route by subject to handleAssigned or handleReminder

    processContactLogCreated(msg):
      1. Unmarshal ContactLogCreatedEvent
      2. Mark voter as contacted: MarkTaskVoterContacted(voterID)
      3. GetTasksLinkedToVoter(companyID, voterID, turfID)
      4. For each task: RecalculateTaskProgress(taskID)
      5. msg.Ack()

    handleAssigned(msg):
      1. Unmarshal TaskAssignedEvent (taskID, userID, taskTitle)
      2. Get device tokens for user (via eden queries or adapter)
      3. If dispatcher != nil && dispatcher.IsEnabled(): dispatcher.SendPush(tokens, "New Task Assigned", taskTitle, data)
      4. msg.Ack()

    handleReminder(msg):
      1. Unmarshal TaskReminderEvent (taskID, userIDs, taskTitle, dueDate)
      2. For each userID, get device tokens
      3. If dispatcher enabled: SendPush with "Task Due Soon" + title + due date
      4. msg.Ack()

    Stop(): cancel()
    ```
